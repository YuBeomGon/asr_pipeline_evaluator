# ASR Pipeline Evaluator

Use this local Claude Code skill when comparing multiple ASR serving pipeline candidates.

## Inputs

Expected candidate folders:

```text
outputs/spec-kit/
outputs/openspec/
outputs/gsd/
outputs/bmad/
```

Each candidate should contain a complete or partial ASR serving pipeline implementation.

## Core evaluation command

```bash
python .claude/skills/asr-pipeline-evaluator/scripts/run_all.py \
  --targets outputs/spec-kit outputs/openspec outputs/gsd outputs/bmad \
  --labels spec-kit openspec gsd bmad \
  --manifest .claude/skills/asr-pipeline-evaluator/assets/baseline/test_fixtures/manifest.jsonl \
  --rubric .claude/skills/asr-pipeline-evaluator/assets/baseline/rubric.yaml \
  --out reports
```

Add `--run-pytest` only when each candidate has a usable Python test environment.

## Evaluation principles

- Compare outputs, not framework reputation.
- Do not change the rubric between candidates.
- Do not let one candidate's code influence another candidate's generation.
- Use the same baseline requirements and API contract for all candidates.
- Treat static score as a first-pass signal, not as the final truth.
- Always include human review for ASR architecture, CER correctness, and production readiness.

## Human review focus

After automated scoring, review:

1. Does the ASR backend abstraction make model replacement easy?
2. Is CER calculation explicit and reproducible?
3. Is transcript normalization defined separately from audio preprocessing?
4. Are `/transcribe`, `/healthz`, and `/metrics` implemented clearly?
5. Are failure modes, timeouts, and logging handled?
6. Is the candidate runnable locally with a mock backend?
7. Is the system extensible to real Whisper/Triton/remote-ASR backends?
